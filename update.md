# پرامپت توسعه — سیستم مانیتورینگ سلامت سرویس‌ها روی Cloudflare Workers

## نقش تو
یک **سیستم مانیتورینگ وضعیت سرویس‌ها (Status Page)** با **پنل ادمین** روی **Cloudflare Workers** از قبل نوشته و در حال کار است. این یک پروژه از صفر نیست — تو باید کدبیس موجود را تغییر بدهی، نه بازنویسی کنی.

**قبل از هر تغییری:**
1. کل ساختار پروژه (فایل‌ها، توابع، روت‌ها، اسکیمای KV/D1، محتوای فعلی `wrangler.toml`) را بررسی و درک کن.
2. کنوانسیون‌های کد موجود (نام‌گذاری، ساختار پوشه‌ها، الگوی نوشتن روت‌ها، استایل CSS فعلی، نحوه مدیریت state در فرانت‌اند) را شناسایی کن و در تغییراتت همان سبک را رعایت کن — کد نباید معلوم باشد دو نفر متفاوت آن را نوشته‌اند.
3. برای هر یک از ۱۲ مورد زیر، مشخص کن که آیا بخشی از آن از قبل پیاده‌سازی شده (و فقط باید تکمیل/اصلاح شود) یا کاملاً جدید است.

**قوانین حین تغییر:**
- فقط بخش‌های لازم را تغییر بده؛ کد و فایل‌هایی که به این تغییرات مربوط نیستند دست‌نخورده بمانند.
- هیچ ویژگی یا مسیر فعلی که کار می‌کند را خراب نکن. اگر تغییری با بخشی از کد موجود تداخل دارد (مثلاً ساختار داده مانیتورها باید تغییر کند)، یک استراتژی migration بنویس تا داده‌های فعلی از بین نرود.
- بعد از هر مرحله (هر بند از ۱۲ مورد)، خلاصه کوتاهی از فایل‌های تغییر‌یافته و دلیل تغییر بده تا قابل ریویو باشد؛ لازم نیست منتظر تأیید بمانی مگر جایی که ابهام واقعی وجود دارد.
- اگر برای پیاده‌سازی یک بند نیاز به تصمیمی است که ساختار فعلی پروژه را محدود می‌کند (مثلاً از KV استفاده شده ولی یک ویژگی به رابطه‌ای‌بودن داده نیاز دارد)، قبل از تغییر بزرگ در معماری، آن را اعلام کن.

## استک فنی (فرض بر این است که از قبل همین‌ها استفاده شده‌اند، مگر خلاف آن در کد دیده شود)
- Cloudflare Workers (یا Pages + Functions) به‌عنوان بک‌اند
- Cloudflare KV (یا D1) برای ذخیره‌سازی تنظیمات، مانیتورها، تاریخچه چک‌ها و اطلاعیه‌ها
- `wrangler.toml` برای پیکربندی build/deploy
- فرانت‌اند: HTML/CSS/JS یا فریم‌ورک سبک فعلی پروژه

---

## ۱. مسیر پنل ادمین قابل تنظیم از wrangler.toml
- دکمه/لینک ورود به داشبورد ادمین از صفحه اصلی (Public Status Page) **کاملاً حذف شود**.
- مسیر پنل ادمین باید از طریق یک متغیر محیطی در `wrangler.toml` تعریف شود، مثلاً:
  ```toml
  [vars]
  ADMIN_PANEL_PATH = "/manage-x7k9"
  ```
- در کد، این مقدار از `env.ADMIN_PANEL_PATH` خوانده شود و روتر بر همان اساس مسیر ادمین را رجیستر کند (اگر مقدار ست نشده بود، یک مقدار پیش‌فرض تصادفی/غیرقابل‌حدس در کد fallback شود، نه یک مسیر ساده مثل `/admin`).
- اگر کسی مسیر پیش‌فرض `/admin` را درخواست کند، باید 404 برگردد (نه ریدایرکت، چون ریدایرکت مسیر واقعی را لو می‌دهد).

## ۲. مدیریت امن متغیرها با `wrangler secret`
- هرچه «حساس» است (رمز عبور ادمین یا هش آن، JWT secret برای سشن ادمین، توکن ربات تلگرام برای بکاپ، هر API Key بیرونی) باید با `wrangler secret put` تنظیم شود، **نه** در بخش `[vars]` فایل `wrangler.toml` (چون آن بخش در کنترل نسخه commit می‌شود و امن نیست).
- در مستندات پروژه (README) لیست کامل secret های موردنیاز و دستور دقیق برای ست کردن هرکدام را بنویس، مثلاً:
  ```
  wrangler secret put ADMIN_PASSWORD_HASH
  wrangler secret put SESSION_SECRET
  wrangler secret put TELEGRAM_BOT_TOKEN
  wrangler secret put TELEGRAM_CHAT_ID
  wrangler secret put TELEGRAM_BACKUP_CHAT_ID
  ```
  توجه: `TELEGRAM_CHAT_ID` مقصد نوتیفیکیشن‌های وضعیت (داون/آپ شدن سرویس‌ها) است و `TELEGRAM_BACKUP_CHAT_ID` مقصد **مجزا** برای ارسال فایل بکاپ است — این دو نباید در یک چت/کانال با هم ترکیب شوند، حتی اگر ادمین هر دو را روی یک عدد یکسان تنظیم کند (باید دو فیلد کاملاً جدا در پنل ادمین باشند).
- رمز عبور ادمین هرگز به‌صورت plain-text ذخیره نشود؛ از هش (bcrypt/argon2 یا حداقل PBKDF2 قابل اجرا در Workers) استفاده شود.
- سشن ادمین با JWT امضاشده (HMAC با `SESSION_SECRET`) و httpOnly + Secure cookie پیاده‌سازی شود؛ منقضی‌شدن سشن و rate-limit برای صفحه لاگین (جلوگیری از brute-force) اضافه شود.

## ۳. برند سفارشی به‌جای عنوان ثابت
- عنوان ثابت `# Monitoring Overview` حذف شود.
- در پنل ادمین امکان تنظیم موارد زیر فراهم شود:
  - نام برند (متن)
  - آپلود/URL لوگوی برند
- در صفحه اصلی، لوگو سمت چپ و نام برند کنار آن نمایش داده شود (به‌جای عنوان ثابت قبلی). اگر لوگو تنظیم نشده بود، فقط متن برند نمایش داده شود.

## ۴. روش‌های بررسی سلامت سرویس: GET + Globalping با انتخاب ریجن
- علاوه بر بررسی فعلی (ارسال درخواست HTTP GET مستقیم از خود Worker)، قابلیت بررسی از طریق **Globalping API** (`https://api.globalping.io`) اضافه شود.
- در تنظیمات هر مانیتور (یا سراسری در تنظیمات ادمین)، ادمین بتواند:
  - روش بررسی را انتخاب کند: `Direct GET` یا `Globalping`
  - در حالت Globalping، یک یا چند **ریجن/لوکیشن** (مثل کشور، قاره یا شهر خاص طبق مستندات Globalping) را از یک لیست/انتخابگر مشخص کند
- نتیجه بررسی Globalping (شامل کد وضعیت، زمان پاسخ، و ریجنی که تست از آن انجام شده) در تاریخچه ذخیره و در UI نمایش داده شود.
- چون Globalping API غیرهمزمان است (ابتدا یک measurement ایجاد و بعد باید نتیجه را poll کرد)، این جریان به‌درستی با تایم‌اوت و مدیریت خطا پیاده‌سازی شود.

## ۵. پیام‌های اطلاع‌رسانی «داون شدن» قابل‌تنظیم از پنل ادمین
- متن پیام/اعلان زمانی که یک سرویس داون تشخیص داده می‌شود، هاردکد نشود.
- در پنل ادمین یک بخش «قالب پیام‌ها» اضافه شود که ادمین بتواند متن پیام داون‌شدن و متن پیام بازگشت (Recovery) را ویرایش کند، با پشتیبانی از متغیرهای جایگزین‌شونده مثل `{{service_name}}`، `{{time}}`، `{{status_code}}`.

## ۶. نمایش/عدم‌نمایش آدرس سلامت سرویس
- برای هر مانیتور یک toggle در پنل ادمین باشد: «نمایش URL مانیتور به کاربر عمومی» (روشن/خاموش).
- اگر خاموش بود، در صفحه عمومی فقط نام سرویس و وضعیت نمایش داده شود، بدون فاش‌شدن آدرس واقعی.

## ۷. تم دارک/لایت با تشخیص خودکار از دستگاه کاربر
- تم بر اساس `prefers-color-scheme` مرورگر کاربر به‌صورت خودکار تنظیم شود.
- یک سوییچر دستی هم در صفحه اصلی باشد که انتخاب کاربر را override کند و در `localStorage` ذخیره شود.

## ۸. زمان آخرین بررسی + بازه بروزرسانی قابل‌تنظیم
- برای هر مانیتور، «آخرین بار بررسی‌شده در» (timestamp نسبی مثل «۲ دقیقه پیش») نمایش داده شود.
- در تنظیمات پنل ادمین، بازه زمانی بررسی خودکار (مثلاً هر ۱، ۵، ۱۰، ۳۰ دقیقه) قابل تنظیم باشد. این بازه با **Cloudflare Cron Triggers** در `wrangler.toml` پیاده‌سازی شود:
  ```toml
  [triggers]
  crons = ["*/5 * * * *"]
  ```
  و مقدار انتخاب‌شده در پنل ادمین باید یا این cron را (در صورت امکان از طریق API دیپلوی) بروزرسانی کند یا در کد Worker به‌عنوان یک آستانه چک شود که آیا زمان بررسی مجدد رسیده یا نه.
- صفحه عمومی هم به‌صورت client-side (polling با `fetch` هر چند ثانیه، بدون نیاز به رفرش کامل صفحه) وضعیت را بروزرسانی کند.

## ۹. ظاهر مدرن
- طراحی مینیمال، فاصله‌گذاری تمیز، کارت‌های وضعیت با انیمیشن ملایم (fade/pulse برای وضعیت آنلاین/آفلاین)، فونت خوانا، و طراحی واکنش‌گرا (موبایل/دسکتاپ) پیاده‌سازی شود. از الگوهای رایج status-page های مدرن (مثل کارت‌های گرید، badge وضعیت رنگی، نمودار uptime روزانه به شکل نوار) الهام بگیر.

## ۱۰. سیستم اطلاعیه/اعلان اختلال (Incident Banner)
- در پنل ادمین امکان ثبت اطلاعیه (عنوان، توضیح، سطح: اطلاع‌رسانی/هشدار/بحرانی، فعال/غیرفعال، زمان شروع و پایان اختیاری) اضافه شود.
- اطلاعیه‌های فعال دقیقاً **بین بخش برند (لوگو/نام) و لیست مانیتورها** در صفحه اصلی نمایش داده شوند، نه بالای صفحه و نه پایین.
- چند اطلاعیه هم‌زمان قابل نمایش باشند (به‌ترتیب زمان ثبت یا سطح اهمیت).

## ۱۱. بکاپ‌گیری کامل از تنظیمات (با گزینه ارسال به تلگرام)
- در پنل ادمین دکمه «دریافت بکاپ» باشد که یک فایل JSON شامل تمام داده‌های سیستم (تنظیمات برند، لیست مانیتورها، قالب پیام‌ها، اطلاعیه‌ها، تنظیمات تم و بازه بروزرسانی) تولید و برای دانلود ارائه دهد.
- امکان **Restore** از فایل بکاپ هم اضافه شود (آپلود JSON → بازنویسی تنظیمات، با تأیید قبل از اعمال).
- یک گزینه در تنظیمات: «ارسال خودکار بکاپ به تلگرام» (روشن/خاموش) + بازه زمانی (مثلاً روزانه/هفتگی).
- این ارسال باید به یک **مقصد کاملاً جدا و مستقل از چت نوتیفیکیشن وضعیت سرویس‌ها** انجام شود:
  - برای نوتیفیکیشن داون/آپ شدن سرویس‌ها: `TELEGRAM_CHAT_ID` (همان که از قبل برای این منظور استفاده می‌شود)
  - برای ارسال بکاپ: `TELEGRAM_BACKUP_CHAT_ID` (secret جدید، مستقل)
  - از همان `TELEGRAM_BOT_TOKEN` می‌توان برای هر دو استفاده کرد، ولی چت مقصد باید متفاوت باشد تا فایل بکاپ در همان کانال هشدارهای وضعیت سرویس‌ها گم/مخلوط نشود.
- در UI پنل ادمین، فیلد «Chat ID مقصد بکاپ» جدا و با توضیح واضح (مثلاً: «این چت با چت نوتیفیکیشن‌های وضعیت متفاوت است») از فیلد «Chat ID نوتیفیکیشن» نمایش داده شود تا ادمین به‌اشتباه یکی نداند.
- فایل بکاپ به‌صورت document (نه فقط متن) به `TELEGRAM_BACKUP_CHAT_ID` ارسال شود. این کار می‌تواند با همان Cron Trigger موجود یا یک cron جداگانه انجام شود.

## ۱۲. پالت رنگی برند قابل‌تنظیم
- تم دارک و لایت باید بر پایه سه‌تایی رنگ‌ها ساخته شود: **سفید، مشکی، و رنگ برند**.
- در پنل ادمین یک **Color Picker** باشد تا ادمین رنگ برند (accent color) را انتخاب کند؛ این رنگ به‌صورت یک CSS variable (مثل `--brand-color`) در کل UI (دکمه‌ها، badge های وضعیت مثبت، لینک‌ها، هایلایت‌ها) اعمال شود، در هر دو حالت دارک و لایت، بدون نیاز به تغییر رنگ‌های خنثی (سفید/مشکی) که پس‌زمینه و متن اصلی را می‌سازند.
